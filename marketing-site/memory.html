continuity / memory

Five layers of memory, written every task.

Claude forgets at the end of every conversation. The dotfiles' memory system is the answer to that - a stack of progressively-durable layers, each with its own write cadence and lifespan, indexed in one file you actually read.

The stack

Five layers. Each one outlives the one above it.

Memory isn't a single store. The dotfiles use five distinct layers, each tuned to a different question: "what am I doing right now?", "what did we learn this week?", "what about this project?", "what about this user?", "what does Claude need to know everywhere?"

  1. 01
    ephemeral

    In-session task tracking

    TaskCreate, TaskUpdate, TaskList. Lives only as long as the current conversation. Shows up in the spinner so you can see what Claude is doing right now. Vanishes when the session ends.

    Lifespanthis conversation Written byClaude, automatically Read byyou, in the UI

  2. 02
    per-task

    File-based session memory

    After every discrete task - a fix, a refactor, a copy change, a decision - Claude writes a short note to .claude/memory/session_YYYY-MM-DD_topic.md. The team rules treat this as mandatory. Batching is a failure mode.

    These files survive context compactions, cross-session restarts, and machine swaps. They're plain markdown with YAML frontmatter and they live in the repo, so they ride along with the code.

    Lifespanforever (or until you delete) Written byClaude, per task Read byfuture Claudes, at session start

  3. 03
    index

    MEMORY.md - the table of contents

    One file per memory directory, one line per memory file. The hook system loads MEMORY.md into every session start, so Claude opens the conversation already knowing the shape of what's been written before.

    Entries are short by rule: under 200 characters, link plus a hook sentence. Detail belongs in the topic file, not the index. The system enforces a 24KB limit; over that the index is truncated and you get a warning.

    Lifespanmatches the topic files it points at Written byClaude, alongside each topic write Read byevery session, automatically

  4. 04
    cross-machine

    Dotfiles git sync

    The entire .claude directory ships in the dotfiles repo. Pull on a new machine and Claude opens with the same memory you wrote on the last one. No upload, no cloud, no auth - just git clone and a symlink.

    When a memory write notes "this happened on the laptop", you can pull on the desktop the next day and Claude already knows. The dotfiles bring their own continuity.

    Lifespanas long as the repo exists Synced byregular git push / pull Read byevery machine you clone to

  5. 05
    global

    Auto-memory project store

    A second memory directory at ~/.claude/projects/<path>/memory/. Separate from the repo memory. Holds durable user-level facts, validated decisions, and behavioral feedback that should follow Claude across projects.

    Five frontmatter types live here: user, feedback, project, decision, reference. Each has its own writing protocol - feedback memories include a "Why" line, decision memories list rejected alternatives, project memories convert relative dates to absolute.

    Lifespanspans the user, not the repo Written byClaude, when patterns emerge Read byevery conversation on this machine

The shape

Frontmatter, typed. A graph, not a pile.

Every memory file opens with YAML frontmatter that tells future-Claude what kind of memory this is and how it relates to others. The system reads these and walks the relationships when the topic comes up. 

---
name: session-2026-05-25-sprint12-closed
description: Sprint 12 closed - first clean dogfood, 8/8 flows
type: project
relates_to:
  - session_2026-05-25_sprint12_design.md
  - feedback_chief_architect_autonomous_dogfood_loop.md
supersedes:
  - session_2026-05-24_dogfood_first_attempt.md
---

Human collaborator: Jonah. Executed autonomously...

Five types.

user - facts about the human: role, responsibilities, preferences, knowledge depth.

feedback - corrections AND validated approaches. Each one carries a Why and a How-to-apply.

project - state of in-flight work: who is doing what, by when, why it matters. Dated.

decision - resolved trade-offs. Names the chosen approach, the rejected alternatives, the conditions that would warrant revisiting.

reference - pointers to external systems: Linear projects, Grafana dashboards, doc folders.

Three relationships.

relates_to - a list. The memory shares topic or context with these other entries. Not symmetric by default.

supersedes - a single pointer. This memory replaces an older one. When written, the old file gets a matching superseded_by back-link.

superseded_by - the back-link. Tells you a file is historical and points at what replaced it.

The result is a small graph, not a pile of files. When Claude reads a topic, it can walk the links to find related context without dumping the whole memory directory into the conversation.

The discipline

When you write matters more than what you write.

The team rules treat memory writing as mandatory, frequent, and never permission-gated. Failures are mistakes you fix; they are not optional.

Per task, not per session.

The moment a fix lands, a copy change ships, or a decision gets made - before the next thing, before responding to the user - the relevant session memory gets a one-line update. Batching is what loses work when a session ends unexpectedly.

No permission gate.

The settings file grants standing write permission to every memory path. Claude does not ask "should I save this?" - it just saves. The team rules call out the pause-to-confirm pattern as a configuration bug, not a question.

Self-analysis after every correction.

When Claude misses a step or breaks a rule, the response isn't an apology - it's a two-question analysis written into a feedback memory: why did this happen, what shortcut did I take? The fix is downstream of understanding the failure mode.

Write-time link check.

When a memory is written or updated, Claude scans the MEMORY.md index already in context for related entries and adds relates_to links where they fit. If this memory replaces an older one, the supersedes field gets set and the old file is updated with a matching back-link. The graph stays consistent without a separate cleanup pass.

Why bother

Because the alternative is starting over every morning.

Without this layer, Claude opens every conversation cold - no idea what was decided yesterday, which approaches failed, which user preferences are settled. The memory system is the difference between a tool you teach once and a tool you teach every time.
