continuity / beats

It remembers so you don't have to.

Claude forgets at the end of every conversation. Beats is the answer: a running record of what got done, what was decided, and why. The next time you open the project, Claude reads that record back before it touches anything.

The whole move

You do not run a command. It just keeps the record.

The single most useful thing to know about Beats is that you do nothing to use it. As you work, Claude keeps notes. The next time you open the project, it reads them back. There is no command to run, no note to file, no checkbox to remember.

Like a colleague who takes notes.

Finish a fix, and a short entry lands in the record: what changed, what broke, how it was resolved. Make a call between two approaches, and the reasoning gets written down alongside the choice. None of this interrupts you.

It matters most across time.

A week later, when you ask Claude to pick up where you left off, it is not guessing from the code or the commit log. It is reading its own account of the work: the dead ends, the decisions, the things that turned out to be true. Recording happens as part of doing the work; reading happens at the start of every session.

Reading the record

Plain text files, and one index that loads them all.

The record is a folder of plain Markdown files inside your project, at .claude/memory/. Most are session notes, one per topic per day. You can open any of them in any editor and read them like a journal entry, because that is what they are.

MEMORY.md, the table of contents.

One file sits above the rest: a one-line-per-entry index of every note, newest first. At the start of a session Claude reads the index, then reads the notes it points to. That is how a fresh session catches up on months of work in seconds.

Just ask.

You never need to open the folder. The natural way to read the record is to ask: "What did we work on last?", "Why did we choose this approach?", "What have we already tried for this bug?" The answer comes from the notes, not from a guess and not from the commit history.

Across machines

The record lives with the project, so it travels with it.

The notes are not stored on your laptop. They live inside the project, in the same place as the code. When the project moves, the record moves with it - nothing to sync by hand, no separate database, no per-machine setup. If you have the project, you have its memory.

A shared record, not a private one.

Open the same project on a different machine, or hand it to a teammate, and the full history of decisions shows up there too. Each note records who did the work, so you can see whose hands last touched a feature. Two people on the same project read from and add to one continuous account, not two that quietly drift apart.

Treat it like the code.

Because the record lives with the project and is visible to anyone who has it, notes are written in plain, professional language. It is part of the repo, and it reads like it.

Steering what it keeps

Most of the time it builds itself. When you want to steer, you say so.

If you tell Claude something that should outlast the conversation, say it plainly: "Remember that we never use broad CSS rules here," "Note for next time: the dev server runs on port 4831." It becomes a note and comes back when the topic is relevant. If something it wrote is wrong, tell it, and the note gets updated or replaced.

The kinds of notes it keeps.

About you - how you like to work, preferences that hold across the whole project. Feedback - a correction or standing instruction, so the same mistake is not repeated. Project state - what is happening right now: what shipped, what is wired up, what is in progress. Decisions - a choice between real alternatives, with the reasoning. References - durable facts worth keeping: ports, file locations, how a tool is set up.

State versus decision.

The distinction that matters most: a project note says what is true now; a decision note says what you chose and why. That reasoning stays useful long after the state has moved on. When you find yourself facing the same fork twice, the decision notes tell you whether the original reasoning still holds.

It is not the only thing that writes.

Sidecoach keeps its own entries in the same record when it runs a design pass, so the reasoning behind a visual change lands next to the rest of the work. The record is the shared substrate the rest of the toolkit reads from and writes to.

The discipline

When you write matters more than what you write.

The team rules treat recording as mandatory, frequent, and never permission-gated. The moment a fix lands or a decision gets made - before the next thing - the relevant note gets a one-line update. Batching is what loses work when a session ends unexpectedly.

No permission gate.

Claude does not ask "should I save this?" - it just saves. The pause-to-confirm pattern is treated as a configuration bug, not a question.

Self-analysis after a correction.

When Claude misses a step or breaks a rule, the response is not an apology - it is a short analysis written into the record: why did this happen, what shortcut did I take? The fix is downstream of understanding the failure.

A graph, not a pile.

When a note is written, Claude links it to related entries and, if it replaces an older one, marks the old file superseded. The record stays a small, navigable graph instead of a heap of files - so reading one topic surfaces its context without dumping the whole folder into the conversation.

Why bother

Because the alternative is starting over every morning.

Without this layer, Claude opens every conversation cold - no idea what was decided yesterday, which approaches failed, which preferences are settled. Beats is the difference between a tool you teach once and a tool you teach every time.
