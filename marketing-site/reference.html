improv

A two-pane visual editor that watches your live UI. Direct manipulation in the browser, no synthetic events, your dev server stays the source of truth.

Install

./install.sh --only improv

The installer wires the editor extension, the MCP server registration in ~/.claude/settings.json, and the watch loop. Symlinks the source into ~/.claude/skills/improv/ so future dotfiles pulls pick up upstream changes.

How it works

Two surfaces: the browser preview (a regular iframe pointed at your dev server) and the property panel (an inspector docked to the right edge). The editor extension lights up an element on click and translates property-panel writes into live CSS through your dev server's hot module reload.

The MCP server exposes a small tool set to Claude: get_selection, apply_changes, watch, respond. These are what let Claude see what you're staring at and act on it without a screenshot round-trip.

Commands

  • improv start - boots the preview pane and registers the watch loop with the running session
  • improv stop - tears down the loop and unregisters the watch
  • improv status - prints the current selection, queue length, and connection state

Posture

  • No synthetic events - clicks are real clicks, keypresses are real keypresses
  • No imposed design system - reads your DESIGN.md tokens, falls back to free input with a console warning
  • No remote - the editor runs in your browser against your dev server; nothing leaves the machine
  • Selection survives context compactions and session restarts

sidecoach

An opinionated design orchestrator. Eight craft flows, 22 verb commands, 159 validators across ten design domains.

Install

./install.sh --only sidecoach
# Add PRODUCT.md and DESIGN.md to your project root
/sidecoach craft .

The craft chain

When you run /sidecoach craft against a project, the chain executes in dependency order:

  • flowA_brand_verify - reads PRODUCT.md, resolves register (brand vs product), caches design laws
  • flowB_component_research - pulls component patterns vetted against AI-slop heuristics
  • flowE_motion_patterns - researches easing curves, exponential-only, with six reduced-motion strategies
  • flowF_design_tokens - validates DESIGN.md across all seven design domains
  • flowG_component_implementation - generates token-backed component specs with eight interaction states
  • flowH_motion_integration - wires motion into components with stack-aware templates
  • flowI_accessibility - seven WCAG 2.1 AA domain audits with a screen reader testing plan
  • flowJ_tactical_polish - sixteen-point refinement pass

Verb commands

Twenty-two slash commands covering the full design pipeline:

/sidecoach craft     /sidecoach polish    /sidecoach audit
/sidecoach critique  /sidecoach shape     /sidecoach typeset
/sidecoach layout    /sidecoach adapt     /sidecoach clarify
/sidecoach optimize  /sidecoach harden    /sidecoach onboard
/sidecoach extract   /sidecoach live      /sidecoach teach
/sidecoach document  /sidecoach help

Validator framework

159 rules across the polish standard (22) and extended domains (137). Each validator has a soft-fail mode that records findings in the build report under the relevant domain's letter grade.

Posture

  • No fabricated icons - SVG paths must come byte-for-byte from a registered library source
  • No hardcoded colors - every CSS value resolves through a DESIGN.md token
  • No "request a demo" - the category-reflex detector watches for AI-template phrases
  • No silent failures - the chain doesn't half-finish; missing prereqs raise structured errors

memory

A layered, file-based memory system. Five durability levels, typed frontmatter, a small relationship graph, indexed in a file you actually read.

Layers

  • TaskCreate / TaskUpdate - in-session ephemeral task tracking, lives for one conversation
  • .claude/memory/*.md - per-task notes, persistent in the repo, written after every discrete change
  • .claude/memory/MEMORY.md - one-line index loaded at every session start
  • Dotfiles git sync - the entire .claude directory ships with the repo
  • ~/.claude/projects/<path>/memory/ - the auto-memory store with five frontmatter types

Frontmatter types

  • user - facts about the human collaborator
  • feedback - corrections AND validated approaches with Why + How-to-apply
  • project - in-flight work state, dated absolutely
  • decision - resolved trade-offs with rejected alternatives and revisit conditions
  • reference - pointers to external systems (Linear, Grafana, etc.)

Relationship fields

  • relates_to - list of memories sharing topic or context
  • supersedes - this memory replaces an older file (which gets a matching superseded_by)
  • superseded_by - the back-link, set when this memory is replaced

The discipline

  • Write per task, not per session - batching loses work when the session ends unexpectedly
  • No permission gate - settings grant standing write access; never ask "should I save this?"
  • Self-analysis after every correction - feedback memories include the failure mode, not just the fix
  • Write-time link check - update relates_to and supersedes as the file is written, not in a separate pass

cmux

A multi-pane terminal surface with a built-in browser pane and markdown renderer. The visual-verification path for any UI work.

Install

./install.sh --only cmux

Browser pane

Each project has its own cmux browser surface, declared in the project's memory directory. Screenshot, navigate, click - the verification commands round-trip through a real headless browser.

cmux browser --surface <id> screenshot --out /tmp/page.png
cmux browser --surface <id> navigate "http://localhost:3000"
cmux browser --surface <id> snapshot --interactive

Markdown panes

Open documents inline rather than launching an editor. cmux renders markdown natively in a dedicated surface.

cmux open /absolute/path/to/document.md

Why it matters

cmux is the surface where the verification protocol gets enforced. Screenshots saved to /tmp get tracked by a hook so they can't be claimed-verified without being read.

voice

Whisper.cpp transcription plus OpenAI TTS. Voice memos in, voice notes back. Built around the Discord bridge but works anywhere.

Install

./install.sh --only voice-input voice-output

Installs whisper.cpp locally, downloads the ggml-base.en model, registers the OpenAI TTS MCP server, and symlinks the ~/.claude/transcribe CLI.

Transcribe an audio file

~/.claude/transcribe /path/to/audio.ogg
# Stdout: the transcript
# Stderr: ffmpeg diagnostics

Supports OGG/Opus (Discord voice memos), m4a (iOS), mp3, flac, wav. Auto-converts to 16 kHz mono PCM via ffmpeg before whisper.

Speak a response

When voice output is unmuted (controlled by the ~/.claude/.voice-enabled flag), the mcp__voice-output__speak tool generates spoken audio. The SessionStart voice-mandate hook injects the active mandate per turn so Claude knows whether to call it.

Mid-session toggle

# In the chat
voice on
voice off

discord

A state-aware Discord bot bridge. DM Claude from your phone, pair per user, no group chats. The launcher's prompt adapts to whether you have a token, paired users, both, or neither.

Onboard a new machine

bash ~/.claude/discord-onboard.sh

Walks you through either "I already have a Discord bot" (prompts for token) or "Walk me through making a new one" (numbered Developer Portal steps). Stores the token in macOS Keychain.

Pair a user

Start a Claude session with the Discord channel attached. DM the bot. It returns a 6-character pairing code. Run /discord:access pair <code> in the terminal session. The bot writes a one-shot signal file at ~/.claude/channels/discord/approved/<userId> containing the chat ID, and re-arms its in-memory allowlist on its next poll.

Voice-on-Discord

Discord voice memos arrive as OGG attachments. The transcribe script ingests them automatically. Claude replies in text or, when voice output is active, attaches a generated OGG voice note.

Recovery

If the bot restarts and the in-memory allowlist drops out of sync with access.json, run bash ~/.claude/discord-onboard.sh --repair. It walks the allowFrom list and prompts for the chat_id of any user missing a marker, then writes the file.

hooks

The enforcement layer. A dozen pre- and post-tool hooks that turn team-rule violations into actual blocks rather than gentle reminders.

Content guards

  • content-guard.sh - blocks emojis, em-dashes, AI-attribution lines, legacy model IDs in file writes
  • bash-guard.sh - same patterns in bash commands plus force-push protection and JS-validation-bypass detection

Discipline reminders

  • memory-nudge.sh - fires after Write/Edit/MultiEdit to non-memory files; reminds the dirty-state memory write
  • verify-manual.sh - clears the screenshot-pending flag when the user types "verified"
  • screenshot-open-mandate.sh - sets a pending flag when a screenshot is captured but not Read
  • validation-guard.sh - blocks synthetic clicks, dispatched events, getComputedStyle probes during validation

Voice and Discord

  • voice-mandate.sh - SessionStart hook that injects voice availability + mute state into context
  • voice-toggle.sh - UserPromptSubmit hook that flips the mute flag on "voice on" / "voice off"
  • resume-guard.sh - SessionEnd hook that blocks cmux auto-resume unless toggled

Override protocol

Hooks block. When a legitimate edit needs to write the exact patterns the hooks reject (documentation that quotes a forbidden phrase, hook source that references its own blockers), ask the user for permission to bypass rather than rephrasing. Some gates can be suppressed for a window via touch ~/.claude/.suppress-fix-gate.

validation-guard

A hook that turns "test like a user, not like a developer" into actual enforcement. Blocks synthetic clicks, dispatched events, private-API invocations, and DOM-state probing during validation.

What it blocks

  • element.click() - synthetic click on a DOM node
  • element.dispatchEvent(...) - any synthesized event
  • obj._privateMethod(...) - private API invocation
  • window.__app.method(...) - application-namespace method calls
  • getComputedStyle, getBoundingClientRect, .scrollTop, .textContent - DOM-state probes used to substitute for a screenshot

What it allows

  • cmux click --selector, type --selector, press --key, screenshot
  • chrome MCP left_click with coordinates, type, key, screenshot
  • Bundle setup (e.g., injecting a script tag a mixed-content page couldn't load) - that's setup, not validation

Escape hatches

None. If a click can't reach what you need (shadow DOM with no exposed selector, element off-screen), stop and tell the user. The guard will block the workaround. The bug you're chasing may be exactly in the event path you'd skip.
