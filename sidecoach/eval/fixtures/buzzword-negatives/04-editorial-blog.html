How we cut our build times by 40 percent

By the platform team, posted April 2, 2025

Last quarter our continuous integration pipeline had crept up to eleven minutes for a single commit. Engineers were context-switching while they waited, and the cost of running the fleet of build machines had roughly doubled year over year. We set a goal to bring the median build back under seven minutes without buying more hardware.

Measuring where the time went

Before changing anything, we added timing spans to each stage of the pipeline and collected a week of data. The install step accounted for nearly half the wall-clock time, mostly because the dependency cache was being invalidated on every run by an unrelated lockfile change.

The slowest stage was not the one anyone suspected. We had been optimizing the compiler flags while the cache quietly missed on every build.

Fixing the cache key

We rewrote the cache key to hash only the lockfile and the toolchain version rather than the entire working tree. Hit rates went from roughly twenty percent to over ninety percent within a day, and the install step dropped from five minutes to under forty seconds on a warm cache.

Parallelizing the test suite

The test stage was running serially on one machine. We split the suite across four workers by historical runtime so each worker finishes at about the same moment. That change alone removed another two minutes from the median.

What we would do differently

The biggest lesson was to measure first. We had spent two earlier sprints tuning things that turned out to be noise. Once we had real timing data, the two changes that mattered took an afternoon to write and have held up for three months.
