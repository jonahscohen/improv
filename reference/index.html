Reference manual

claude-dotfiles

The full reference for the Yes& Claude Code stack. Every component, every hook, every layer of the design pipeline, documented. 

curl -fsSL https://raw.githubusercontent.com/jonahscohen/claude-dotfiles/main/bootstrap.sh | bash

Quick start

One curl installs the ampersand shortcut. Run it from any terminal to launch the component picker. 

ampersand                       # interactive TUI
ampersand --pull                # pull latest from GitHub then TUI
ampersand --preset minimal      # brain + config + memory + skills + nvm
ampersand --only memory         # just one component
ampersand --only memory,skills  # multiple
ampersand --dry-run             # preview without writing
ampersand --yes                 # everything, non-interactive

Presets

PresetComponents included
allEverything (same as --yes)
minimalbrain + config + memory + skills + nvm
noneNothing (useful with --dry-run)

Custom clone locations

The dotfiles can live anywhere on disk. The Ghostty config uses a __DOTFILES_DIR__ placeholder that the installer replaces at install time, so cloning to ~/code/dots, /opt/dots, or anywhere else works. 

CLAUDE_DOTFILES_DIR=~/code/dots curl -fsSL .../bootstrap.sh | bash    # env var
curl -fsSL .../bootstrap.sh | bash -s -- --dir ~/code/dots             # flag

Re-running install.sh from a new clone location refreshes the ampersand function's path automatically. No manual cleanup of the old .zshrc block required.

Onboarding a new Yes& dev

  1. curl -fsSL https://raw.githubusercontent.com/jonahscohen/claude-dotfiles/main/bootstrap.sh | bash
  2. TUI defaults, or --preset all to take everything.
  3. Open a new terminal (or source ~/.zshrc).
  4. Done. Same machine state as everyone else.

House 01

Discipline

Rules in CLAUDE.md are advisory - the model skips them under pressure. Hooks at the tool boundary are mechanical - they fire before the tool runs and block the call. Eighteen hooks across five roles enforce the disciplines that mattered. Each hook traces back to a specific real failure that drove its creation.

The pattern: rule violated -> feedback memory written -> rule violated again -> hook built -> violations stop. This isn't a feature; it's a method. When something fails twice, the next response is a hook.

Refusal hooks (PreToolUse, hard deny)

These hooks intercept and block. The tool call never runs.

MatcherHookPurpose
Bash bash-guard.sh Blocks AI-attribution lines in commits, force-push to main / master, rm against .claude/memory, legacy model IDs, project-specific anti-patterns.
Write|Edit|MultiEdit content-guard.sh Same pattern set inside file content - AI-attribution, emdashes, endashes, emoji unicode ranges, legacy model IDs.
Write|Edit|MultiEdit memory-approve.sh Auto-approves memory-path writes so they don't need permission prompts regardless of permission mode.
mcp__voice-output__speak voice-gate.sh Denies speak calls when ~/.claude/.voice-enabled is absent. No wasted OpenAI TTS round-trips when muted.
mcp__claude-in-chrome__javascript_tool validation-guard.sh Blocks .click(), synthetic events, getComputedStyle, getBoundingClientRect, .textContent, underscore-prefixed app methods - the DOM shortcuts that fake UI validation without going through real event paths.

Gate hooks (state machines)

These don't block tool calls outright - they set or clear flag files that other hooks consult. The downstream bash-guard blocks git commit while any gate flag is set.

EventHookPurpose
PostToolUse(Write|Edit|MultiEdit) verify-before-done.sh Sets ~/.claude/.needs-verification on edits to .ts/.tsx/.css/.sh/.py files. Clears on cmux screenshot Read, localhost curl, test commands (npm test / vitest / pytest / etc.), external curl probes with port or path, or /tmp/*test*.log Reads.
PostToolUse(Read) verify-clear.sh Clears .needs-verification on Read of a .png (the screenshot was actually opened, not just saved).
UserPromptSubmit verify-manual.sh Recognizes phrases like "looks good" / "ship it" / "approved" and clears verification manually when the user explicitly signs off.
PostToolUse(Bash|...computer) screenshot-open-mandate.sh Sets ~/.claude/.screenshot-pending after a screenshot is saved. bash-guard blocks subsequent cmux ... screenshot and git commit until the flag clears.
PostToolUse(Read) screenshot-open-clear.sh Clears .screenshot-pending when a matching .png path is Read.
PostToolUse(Write|Edit|MultiEdit) second-fix-gate.sh (v2) Warns once when two fixes land on the same file within 10 minutes with .needs-verification still set. Suppressed on purely-additive Edit / MultiEdit (where new_string contains old_string). Manual override via touch ~/.claude/.suppress-fix-gate (30-min auto-expiring).

Nudge hooks (advisory)

These inject reminder text into the agent's context. They don't block tool calls - they just remind. Two of them, kept intentionally limited so they don't become banner-blind noise.

MatcherHookPurpose
Write|Edit|MultiEdit|Bash memory-nudge.sh Injects "PROJECT FILE CHANGED. You are in dirty state. Write memory before responding" after any non-memory file change. Skipped on paths under .claude/memory/ or matching MEMORY.md.
SessionStart reflect-nudge.sh Counts new memory files since the last reflection. When above threshold (default 15, configurable via REFLECT_THRESHOLD), surfaces a "want to run /reflect?" prompt at session start.

Toggle hooks (state flips on chat phrases)

Trigger phraseHookEffect
"voice on" / "voice off" / "mute yourself" / "unmute" voice-toggle.sh Touches or removes ~/.claude/.voice-enabled. Reads by voice-mandate.sh at SessionStart and voice-gate.sh on every speak call.
"auto-resume on" / "off" resume-toggle.sh Touches or removes ~/.claude/.no-auto-resume. Reads by resume-guard.sh at SessionEnd to decide whether to block cmux auto-resume.

Lifecycle hooks (session boundaries)

EventHookPurpose
SessionStart startup-check.sh Loads project + global memory at session start. Status: "Loading memory..." The bridge that gives Claude continuity across sessions.
SessionStart voice-mandate.sh Reads .voice-enabled and injects an active-mandate, muted-notice, or silence (the gate for the voice subsystem).
PreCompact inline command Flushes pending memory before context compresses. Status: "Flushing memory before compact..."
PostCompact startup-check.sh Re-loads memory after compression. Status: "Reloading memory after compaction..."
SessionEnd resume-guard.sh Blocks cmux auto-resume so a stale session doesn't surprise you on next launch (unless .no-auto-resume is unset).

The authoritative inventory plus flag-file registry plus precedence rules lives in the canonical memory file .claude/memory/decision_hook_system_architecture.md.

The verification protocol

Seven rules in CLAUDE.md that gate task completion. Mandatory, no exceptions.

  1. Visual verification. UI changes must be screenshotted and described. "It renders" is not verification.
  2. Interactive verification. Buttons, dropdowns, toggles must be clicked / typed into and re-screenshotted via real input only. No JS shortcuts.
  3. Side-by-side verification. If building from a Figma reference or mockup, compare implementation against source.
  4. Completeness check. Re-read the original request, list every item, confirm each one exists.
  5. No lazy questions. If the user asked for 5 things and Claude built 3, build the other 2 instead of asking "should I continue?"
  6. No false positives. Passing type check is not a passing feature. Verify with eyes.
  7. For non-UI tasks, state a verifiable plan first. Refactors, scripts, CLI changes: write <step> -> verify: <check> before implementing. Verify clauses must be runnable, not "looks right."

Permission posture (deliberate)

claude/settings.json ships with defaultMode: bypassPermissions and skipDangerousModePermissionPrompt: true. Every tool call auto-approves, and Claude Code's own warning on the bypass mode is suppressed.

This is on purpose. The team decided the friction of per-tool prompts outweighs the safety it adds, and the hook layer (above) already blocks the specific patterns that matter. If you want different defaults, change defaultMode in claude/settings.json to default (per-tool prompting) or acceptEdits (auto-approve edits but not bash). Remove skipDangerousModePermissionPrompt if you want Claude Code's own warning to show.

The hook layer stays useful regardless of defaultMode - hooks fire BEFORE the permission prompt would, so they continue blocking forbidden patterns even in fully-prompting mode.

House 02

Memory

Memory turns Claude Code from a stateless code generator into a colleague who remembers what was decided last week and why. It's the most load-bearing thing in this repo.

The three storage layers

Project root memory

<project>/.claude/memory/

The canonical record for that specific project. Session files, feedback files, reference files, decision files - all indexed by MEMORY.md. Committed to the project's git repo like any source. Read by every collaborator's Claude on session start.

Global cross-project memory

~/.claude/memory/

Per-machine, durable across all projects you open with Claude Code. The attribution policy, team-wide feedback, hook verification discipline. In Yes& setups, this directory is populated by symlinks from the dotfiles repo when you tick the memory component, so every Yes& dev's machine has the same baseline.

Per-project global memory

~/.claude/projects/<project-path>/memory/

Automatically written by Claude Code itself for telemetry-style state. Secondary context, read at session start as a fallback when project root memory is missing.

File format and indexing

Every change writes a session memory entry to <project>/.claude/memory/session_YYYY-MM-DD_<topic>.md before Claude responds. Not per-feature - per-task. A CSS fix, a copy change, a refactor decision each get their own entry immediately. Batching is a failure mode the memory-nudge hook catches. 

---
name: <one-line title>
description: <one-line summary used by future Claude to decide relevance>
type: project
---

Collaborator: <name from git config user.name>

# What changed
[bullet list of concrete changes]

# Why
[motivation - the rationale that future-you needs]

# How to apply
[is it live immediately, do other machines need to pull, etc.]

# Files touched
- file1.ext
- file2.ext

Index it in .claude/memory/MEMORY.md as a one-line entry. The index is the load-bearing data structure for memory retrieval - keep it scannable.

Memory types

  • session_YYYY-MM-DD_<topic>.md - what happened during a work session
  • feedback_<topic>.md - durable behavioral rules and learnings
  • reference_<topic>.md - how to operate a tool, where things live (often global memory)
  • decision_<topic>.md - architectural choices with rationale and revisit conditions
  • reflection_YYYY-MM-DD.md - output of the /reflect skill

Lifecycle hooks

Three hooks (described in the Discipline section) make memory concrete across session boundaries:

  • SessionStart runs ~/.claude/startup-check.sh which loads project + global memory into the session's context.
  • PreCompact flushes pending memory entries before Claude's context window compresses.
  • PostCompact re-runs the startup loader so post-compaction Claude has fresh memory loaded again.

The /reflect skill

/reflect spawns five parallel analysis agents against the accumulated memory corpus:

AgentLensWhat it surfaces
Pattern HunterRecurrenceThemes that repeat across sessions; approaches the team gravitates toward.
Tension DetectorContradictionRules that fight other rules; stated preferences vs observed practice.
Gap AnalystAbsenceWhat's missing that you would expect to find; underrepresented memory types.
Drift TrackerEvolutionHow practices have shifted over time; emerging concerns.
Decision ArchaeologistStalenessDecisions whose conditions may have been met; assumptions that no longer hold.

A synthesis agent weaves all findings into a unified narrative with ranked findings, open questions, and recommended actions. The output saves to .claude/memory/reflection_YYYY-MM-DD.md and informs the next round of hooks and rules.

A SessionStart reflect-nudge hook counts new memory files since the last reflection. When the threshold is exceeded (default 15, configurable via REFLECT_THRESHOLD), the session opener nudges you. The user says yes and it runs, no and it drops.

Attribution and collaboration

Every memory entry records a Collaborator: line derived from git config user.name. So when Alice works, her Claude tags her name; when Bob works, his Claude tags his. Memory files commit and push like any source file, so attribution flows through git like any other authorship signal.

Three layers of authorship come out of this:

  • Per memory entry: the Collaborator: line at the top
  • Per memory file: git log and git blame (identical, but ironclad)
  • Per task: the Why: and How: fields explain reasoning so reading a session memory tells you not just what happened but who decided what and why

The assistant stays invisible in the output. No AI-attribution lines in commits, no auto-generated credit comments, no AI authorship anywhere. Humans are named; the assistant isn't.

Project-level CLAUDE.md template

For ironclad enforcement on projects where teammates haven't installed the dotfiles, drop a CLAUDE.md at the project root that re-states the rules. Claude Code reads project-root CLAUDE.md regardless of dotfiles state: 

## Memory Discipline (project-level)

After every discrete change in this project, write a memory entry to
.claude/memory/session_YYYY-MM-DD_<topic>.md before responding to the user.

Include:
- Frontmatter (name, description, type: project)
- A Collaborator: line with the human's name from git config user.name
- A Why: rationale and How: approach for non-trivial decisions
- A list of files touched

Update .claude/memory/MEMORY.md to index the new file.

This applies regardless of whether the developer has Yes&-dotfiles installed.

House 03

Design

Most AI-generated UI looks the same because most prompts ask for the same vague thing. This stack routes design work through seven layered skills, each addressing a different question at a different beat. The /design-build orchestrator (added 2026-05-20) runs them as one coordinated sequence with mandatory QA.

The /design-build orchestrator

Empirically, the design pipeline's individual skills don't auto-trigger reliably on real builds, and the QA triad (impeccable audit / critique / polish) never fires without explicit invocation. The /design-build skill solves both: one explicit invocation, ten phases, two gate checkpoints (strategy + QA findings), mandatory QA.

Trigger only on explicit phrases: /design-build <feature>, "run the design pipeline for X," "kickoff a design build," "use the design stack on X." Do NOT trigger on generic UI keywords like "build a button" - those route to individual skills.

Phase 0 - Pre-flight

Check PRODUCT.md (required) and DESIGN.md (nudge-once if missing). Capture brand voice words, register, anti-references.

Phase 1 - Strategy

Compose direction. Gate checkpoint 1 via AskUserQuestion - approve / edit / reframe.

Phases 2-7 - Build

Research, references catalog grep, typography (if scope), motion (if scope), icons (if needed), build with feel-better rules applied during construction.

Phase 8 - QA triad (MANDATORY)

/impeccable audit (5-dimension technical scan), /impeccable critique (design review via sub-agents), /impeccable polish (final design-system alignment). Cannot be skipped. Gate checkpoint 2 on findings.

Phase 9 - Verification

cmux browser, screenshots, real-input testing (no JS shortcuts).

Phase 10 - Memory

Session memory documenting which phases fired, which skipped (with reason), QA findings, friction points.

1. Strategy - /impeccable

A plugin (impeccable@impeccable) auto-installed via your enabledPlugins. Twenty-three commands. Reads PRODUCT.md and DESIGN.md at the project root before every command.

Key commands across the lifecycle:

  • /impeccable teach - interactive PRODUCT.md authoring
  • /impeccable document - extract existing visual system into DESIGN.md
  • /impeccable shape <feature> - plan only, no code
  • /impeccable craft <feature> - build from scratch
  • /impeccable audit / critique / polish - the QA triad
  • /impeccable animate / colorize / delight / bolder / overdrive - add personality
  • /impeccable quieter / distill - tone down a loud UI
  • /impeccable typeset / layout / adapt / clarify / optimize - fix specific dimensions
  • /impeccable harden - production-ready sweep (errors, i18n, edge cases)
  • /impeccable onboard - first-run flows, empty states, activation
  • /impeccable extract - pull reusable tokens and components
  • /impeccable live - iterate visually in a live browser

2. Research - component-gallery-reference

A bundled skill that has Claude browse component.gallery before building any standard UI component. The site catalogs 60 component types across 95 design systems (Polaris, Carbon, Primer, Spectrum, Material) with 2,672 examples.

Workflow: detect tech stack from package.json, browse the component page filtered by stack, exclude examples tagged "Unmaintained" or "Accessibility issues," inventory the project's design system, synthesize a brief mapping gallery patterns onto project tokens. Then build with three layers: function from the gallery, identity from the project, gap-fills derived from gallery patterns styled with project tokens.

3. Typography - fontshare-reference

A bundled skill that researches typefaces via fontshare.com, Indian Type Foundry's curated open-source catalog. The skill bakes in Impeccable's reflex-reject list - the training-data-default typefaces that get refused as primaries on greenfield work:

Fraunces, Newsreader, Lora, Crimson, Crimson Pro, Crimson Text, Playfair Display, Cormorant, Cormorant Garamond, Syne, IBM Plex Mono, IBM Plex Sans, IBM Plex Serif, Space Mono, Space Grotesk, Inter, DM Sans, DM Serif Display, DM Serif Text, Outfit, Plus Jakarta Sans, Instrument Sans, Instrument Serif.

The skill also flags Fontshare's emerging defaults (General Sans, Cabinet Grotesk, Switzer, Satoshi, Clash Display) so the catalog doesn't become the new Inter.

4. References - design-references + /curate

The personal catalog system. ~/.claude/design-references/ holds one folder per reference (markdown + screenshot). The catalog grows from your eye over time.

/curate is the capture wizard. Five-step interactive flow: source -> auto-tag proposal -> why-interesting body -> slug -> save. Hybrid tagging: strict Category from controlled vocab (list, navigation, command-palette, inline-affordance, etc.); free-form Pattern + Feel.

design-references is the retrieval skill. Auto-triggers on UI build keywords, greps the catalog for matching category / pattern / feel against the task context AND against PRODUCT.md voice words. Surfaces 0-5 references with score >= 3. Stays silent if nothing scores - noisy surfacing destroys trust.

What this layer covers that public catalogs don't: one-off patterns. A particular animation. A particular interaction. A specific brand's distinctive treatment. The patterns that don't fit any standard taxonomy but are worth remembering.

5. Motion - motion-reference (GSAP + Lenis)

A bundled skill shipping canonical patterns for the GSAP + Lenis stack. As of Webflow's acquisition of GreenSock, all formerly-paid GSAP plugins are free - SplitText, MorphSVG, DrawSVG, ScrollSmoother, Inertia, MotionPathHelper. No paywall.

Routes by task: tweens / timelines -> gsap; scroll-driven (pin / scrub / snap) -> ScrollTrigger; smooth-scroll feel -> Lenis; layout transitions -> Flip; SVG path draw -> DrawSVG; text by word / char -> SplitText; drag -> Draggable.

Critical patterns shipped inline: the GSAP + ScrollTrigger + Lenis three-line glue snippet, React useGSAP hook with scope, ReactLenis root provider. Plus the gotchas: SSR, ScrollTrigger.refresh() after dynamic content, Lenis breaks native scrollIntoView, iOS Safari + Lenis + position: fixed quirks.

6. Tokens - DESIGN.md

Google's spec for representing a visual identity to coding agents (google-labs-code/design.md). YAML frontmatter for tokens (colors, typography, rounded, spacing, components with {path.to.token} references), markdown body for rationale. Comes with npx @google/design.md lint for schema validation, WCAG contrast checks, broken-ref detection.

CLAUDE.md mandate: conform to the Google spec, run lint after every write, resolve every error or warning before reporting done. Generated UI references tokens via {path.to.token}, not hex literals - the file stays the source of truth.

7. Tactical - make-interfaces-feel-better

An Anthropic Skill that auto-triggers on UI keywords. Sixteen specific rules with exact values, applied DURING construction (not as a post-pass).

Concentric radius - outer = inner + padding
Optical centering - icons need manual nudge past geometric
Shadows over borders - layered transparent box-shadow
Interruptible animations - CSS transitions for state, keyframes only for staged
Split + stagger enters - semantic chunks, ~100ms each
Subtle exits - small fixed translateY, never full height
Contextual icon swaps - opacity + scale + blur sequence
Font smoothing on root - -webkit-font-smoothing: antialiased
Tabular nums - on every dynamic counter
text-wrap balance on headings; pretty on body
Image outlines - rgba(0,0,0,0.1) light / rgba(255,255,255,0.1) dark, never tinted
scale(0.96) on press - always 0.96
initial={false} on AnimatePresence to skip first-load
Never transition: all - specify exact properties
will-change sparingly - only transform / opacity / filter
40x40 hit areas minimum, no overlap

Peer skills (siblings, not in the main pipeline)

The skills component also bundles four peers that ship alongside but route in independently when needed:

social-media

Platform specs for 13 social platforms (Instagram, YouTube, TikTok, Twitter / X, LinkedIn, Threads, Bluesky, Discord, GitHub, Dribbble, Behance, Product Hunt, Substack). Sizes, safe zones, typography rules. The spec sheet, not the paintbrush.

design-team

Multi-agent design sprints with 16 specialized roles across 4 phases: Research -> Build -> Review -> Revise. For full pages, campaigns, or design system creation. Not for single components.

visual-effects

Generative shader backgrounds (14 types) and 17 stackable post-process FX. Ships actual shader source code as canonical reference implementations - not abstractions.

icon-source

One library per project. Eight approved (Heroicons, Lucide, Tabler, Bootstrap Icons, Phosphor, Material Symbols, Lucide Animated, Heroicons Animated). Verbatim path sourcing - never redraw or "optimize" SVG paths.

PRODUCT.md spec (what Impeccable reads)

PRODUCT.md captures the brand strategy that all design work routes through. Five sections, all load-bearing:

  1. Register. Brand-side or product-side. Affects voice, density, motion budget.
  2. Who this is for. Primary, secondary, tertiary audiences. Concrete, not "everyone."
  3. Brand personality. Three physical-object words. Not "modern" or "elegant" - "engineering-firm tailor," "quiet midnight workshop," "annotated editorial."
  4. Anti-references. The saturated lanes to avoid. Concrete: "not Vercel monochrome," "not SaaS template," "not Klim editorial-typographic."
  5. Strategic principles. 3-5 opinions that show up in every decision. Examples: "discipline before sparkle," "show don't shout."

House 04 - Workflow

The twelve components

Each is independently togglable in the TUI. Defaults are all on. Every component is additive - nothing overwrites your existing config.

brain

Team rules + workflow

Appends team rules (RULES.md) and shared workflow instructions to your CLAUDE.md between marker comments. Your existing content stays. Re-runs detect markers and skip. CLAUDE.local.md for personal overrides gets its own marker block.

Touches: ~/.claude/CLAUDE.md (your content preserved)

config

Hooks + plugins + permissions

JSON-merges safety hooks, memory-write allow patterns, enabled plugins, and marketplace entries into your existing settings.json. Does NOT touch your defaultMode, model, or other preferences. Copies hook scripts to ~/.claude/hooks/ alongside any hooks you already have.

Touches: ~/.claude/settings.json + ~/.claude/hooks/

memory

Memory discipline + loader

Bolts the memory subsystem onto an existing Claude Code: appends Memory Discipline rules to CLAUDE.md between markers, JSON-merges three lifecycle hooks into settings.json, symlinks the loader script. Marker-guarded - re-runs are no-ops, removable cleanly.

Touches: CLAUDE.md, settings.json, startup-check.sh symlink

skills

10 Anthropic Skills + catalog seed

Bundles make-interfaces-feel-better (tactical polish), component-gallery-reference (research), fontshare-reference (typography), motion-reference (GSAP + Lenis), design-build (pipeline orchestrator), curate + design-references (personal catalog), plus four peer skills (social-media, design-team, visual-effects, icon-source). Seeds the empty ~/.claude/design-references/ catalog with the 14-category vocab.

Touches: ~/.claude/skills/ + ~/.claude/design-references/

statusline

Custom prompt-bar render

The statusLine command in settings.json is tolerant of a missing script, so unticking falls back cleanly to Claude Code's default. Pick if you like the prompt-bar render; skip if you have your own statusline.

Touches: ~/.claude/statusline-command.sh (symlink)

cmux

Split-pane terminal config

cmux is the split-pane terminal that hosts the in-app browser preview Claude uses to verify UI work. The settings symlink keeps your cmux config in sync with the dotfiles repo. Powers cmux browser ... screenshot for visual verification.

Touches: ~/.config/cmux/settings.json (symlink)

nvm

PATH fix for fresh terminals

Fix for the specific issue where Homebrew's nvm sources nvm.sh (making the nvm command available) but doesn't activate a default Node version, leaving claude not in PATH. Appends nvm use default --silent to your .zshrc only if nvm.sh is already sourced. Harmless no-op otherwise.

Touches: ~/.zshrc (marker-guarded line)

ampersand

The ampersand zsh shortcut

Type ampersand from any terminal to re-launch this installer. ampersand --pull pulls latest from GitHub first. Forwards every other flag. Auto-migrates older block formats from prior versions of the installer.

Touches: ~/.zshrc (marker-guarded block)

discord

Discord chat agent launcher

Smart claude wrapper with cold / mid / warm onboarding states. Cold (no bot): offers setup walkthrough or "never ask again." Mid (bot configured, no users paired): jumps to pairing. Warm (paired): 5-second connect prompt, default Yes. Tokens in macOS Keychain, never in repo.

Touches: ~/.zshrc + ~/.claude/ symlinks

voice-input

Local voice-to-text

Brews whisper-cpp and ffmpeg, downloads the ggml-base.en model (~150 MB) into ~/.cache/whisper/, symlinks bin/transcribe to ~/.claude/transcribe. Handles OGG/Opus, m4a, mp3, flac, wav. Local-only - no cloud, no API key.

Touches: brew deps + ~/.claude/transcribe symlink

voice-output

OpenAI TTS responses

Claude speaks short verbal summaries while keeping code and technical detail as text. Requires an OpenAI API key in macOS Keychain. Starts muted - enable with voice-on in any terminal. Three mute controls: in-session, terminal alias, or manual file toggle. Voice subsystem is gated by a SessionStart hook that injects an active-mandate, muted-notice, or silence based on the toggle state.

Touches: MCP server + zsh aliases

reflect

Memory corpus analysis

Adds the /reflect skill (5 parallel analysis agents) plus the SessionStart nudge hook that surfaces when enough new memories have accumulated. Triggers naturally from conversation ("what patterns are you seeing?") or explicitly via /reflect.

Touches: ~/.claude/skills/reflect/ + nudge hook + timestamp file

Customizing settings.json

config JSON-merges hooks, plugins, and permission patterns into your existing ~/.claude/settings.json. It does NOT touch your defaultMode, model, or other preferences.

  • Adding a plugin: edit enabledPlugins in claude/settings.json in the repo and add "<plugin>@<marketplace>": true. Commit, push, ampersand --pull on other machines to re-merge, restart Claude Code.
  • Tightening permissions: edit permissions.allow in your own ~/.claude/settings.json directly. Or change defaultMode from bypassPermissions to default to require approval on tool calls.
  • Adding a hook: append to the appropriate hooks.<event> array in claude/settings.json in the repo. Commit, push, re-run install to re-merge.

Customizing CLAUDE.md

brain appends team rules between marker comments in your ~/.claude/CLAUDE.md. Your own content above and below the markers is preserved.

  • To update the team rules: edit claude/CLAUDE.md in the repo, commit, push, and re-run ampersand --pull to re-append.
  • To remove the team rules: sed-delete from <!-- claude-dotfiles:brain:begin --> to <!-- claude-dotfiles:brain:end -->.
  • To re-add: re-run install.sh --only brain.

Workflow

Architecture

Under-the-hood specifics: how the installer writes to each target, how idempotency works, backup discipline, and multi-location support.

Install strategy by target

TargetStrategyWhy
~/.claude/CLAUDE.mdMarker-guarded appendYour content preserved; re-runs detect markers and skip; uninstall is sed-deletable.
~/.claude/settings.jsonJSON-merge (python3)Your settings preserved; our hooks / plugins / patterns merged in; does not touch defaultMode.
~/.claude/hooks/*.shFile copy from repoHook scripts copied alongside your existing hooks; re-run updates from repo.
~/.claude/memory/*.mdSymlink to repoMemory edits write directly into the repo working tree (Yes& baseline files are versioned).
~/.claude/skills/<skill>/npx-installed or file copySkills versioned by their own repo (npx) or bundled with ours (copy).

Idempotency model

Every section of install.sh is idempotent:

  • Symlinks - make_symlink checks if the target already points where we want; if so, no-op. Otherwise backs up any pre-existing real file, creates fresh.
  • .zshrc appends - marker-guarded with grep checks. If the marker is present, no-op. The ampersand block self-heals: if $REPO_DIR doesn't match the baked path, the block is sed-deleted and re-appended.
  • settings.json JSON-merge - marker-based detection on substrings (hook command paths, plugin names). If detected, no-op. Otherwise python3 reads existing settings.json, adds missing entries, writes back.
  • CLAUDE.md marker-append - marker-guarded on <!-- claude-dotfiles:<component>:begin -->. If present, no-op.
  • Hook file copy - overwrites only our own scripts (same filename), never touches hooks you wrote.
  • npx skills add - idempotent via the skills CLI's own logic.

Backup discipline

Any pre-existing real (non-symlink) file at a target path gets copied to .backups/<timestamp>/<original-path> before the installer overwrites it. Backups are gitignored. To recover from a bad install, walk .backups/ and copy back what you want.

We don't ship an automated uninstaller because it would require us to know which timestamp's backups to use. Manual recovery is safer.

Multi-location support

The dotfiles can be cloned to any path - ~/code/dots, /opt/dots, ~/dotfiles/yes-and. The ampersand shortcut bakes in the install-time $REPO_DIR. If you move the repo, re-running install.sh from the new location triggers the path-drift self-heal that rewrites the shortcut block.

Drift management (source vs installed)

A real architectural detail: ~/.claude/CLAUDE.md is the concatenation of claude/RULES.md + claude/CLAUDE.md + claude/memory-discipline-section.md from the dotfiles repo, assembled at install time. Edits to the installed file directly will be wiped by the next installer run. To make changes durable, edit the source files and re-run ampersand --pull.

Similar with ~/.claude/settings.json - config JSON-merges from claude/settings.json in the repo. The installed file is the merge result, not the source. Edit the source file in the repo, push, pull, and re-run install on each machine.

Operating it

Troubleshooting

"claude: command not found" in fresh terminals

Cause: Homebrew's nvm sources nvm.sh (making the nvm command available) but doesn't activate a default Node version. The cmux claude wrapper is first in PATH, runs when you type claude, looks for the real claude binary elsewhere in PATH, finds nothing because nvm hasn't activated.

Fix: tick the nvm component in the installer. It appends nvm use default --silent to your .zshrc. New terminal (or source ~/.zshrc) and claude should resolve.

ampersand: command not found immediately after install

Cause: Shell functions defined inside install.sh's child process don't escape into the parent shell. Your current zsh hasn't read the updated .zshrc yet.

Fix: source ~/.zshrc once, or open a new terminal window. Subsequent shells read .zshrc at startup so this only happens once per machine.

Permission prompts on every markdown write

Cause: Should not happen with defaultMode: bypassPermissions set, but Claude Code or its harness sometimes escalates anyway.

Fix: claude/settings.json already includes explicit Write(**/*.md), Edit(**/*.md), MultiEdit(**/*.md) allow rules. If you're still seeing prompts, restart your Claude Code session - permissions are loaded at session start. If they persist, broaden the allow list.

gum not installed and I don't want to install it

The TUI degrades gracefully to a numbered text menu if gum isn't available and you decline to brew-install it. Same components, same flags, less polish. Pick numbers to toggle off, press enter to confirm.

Memory entries from a teammate on a different machine

Pull the project: git pull in the project repo. The memory files are in <project>/.claude/memory/ like any other source. Claude reads them at session start.

Fresh install on a new Mac, where do I start 
curl -fsSL https://raw.githubusercontent.com/jonahscohen/claude-dotfiles/main/bootstrap.sh | bash

Take TUI defaults. Open a new terminal. Done.

I want to install without pulling first

Use ampersand (no --pull). Same install process, no git pull step. Useful when you're iterating on the dotfiles locally and don't want to pull until you've finished a series of edits.

My team has its own Claude Code config and I don't want to overwrite it

Every component is additive. brain appends to your CLAUDE.md between markers. config JSON-merges into your settings.json. Nothing is replaced. Just run ampersand and pick what you want.

ampersand --pull --only brain,config,memory,skills

That installs the team rules, hooks / plugins, memory subsystem, and skills. Your existing content in CLAUDE.md and settings.json stays intact.

How do I undo the dotfiles entirely

For marker-appended files (CLAUDE.md): sed-delete from the marker begin to marker end. For JSON-merged settings (settings.json): remove the specific hook / plugin entries that were added. For symlinked files (statusline, memory, cmux): rm them and copy back from .backups/<some-timestamp>/ if needed. For appended .zshrc blocks: sed-delete the marker-guarded ranges.

We don't ship an automated uninstaller because it would require us to know which timestamp's backups to use. Manual is safer.

Hooks fire too often or in the wrong context

Some hooks have known false-positive paths (e.g., memory-nudge firing on Reads, the second-fix-gate firing on sequential additive edits as one coherent change). When this happens during a known multi-step task, touch ~/.claude/.suppress-fix-gate silences the second-fix gate for 30 minutes (auto-expiring).

For persistent misfires, the authoritative hook documentation lives in .claude/memory/decision_hook_system_architecture.md. Tune the matchers in ~/.claude/settings.json if a hook fires on events it shouldn't.

For Yes& devs

Contributing

Adding a new component

  1. Add the key to KEYS=(...) in install.sh and add a TITLE + DESC in the same arrays. Bump the PICKS array length.
  2. Add an apply block (if picked <key>; then ... fi) in the appropriate numbered section. Renumber subsequent sections to stay sequential.
  3. If the component modifies ~/.zshrc, use a marker-guarded append pattern (see nvm or ampersand blocks for examples). Always include path-drift self-heal logic if the block bakes in $REPO_DIR.
  4. If the component modifies ~/.claude/settings.json JSON-style, use the python3 stdlib merge pattern from the memory component. Marker-detection is mandatory.
  5. Update --help valid keys list and the post-install summary's picked <key> && echo "..." line.
  6. Update the README + this reference manual's component grid.
  7. Write a session memory entry: .claude/memory/session_YYYY-MM-DD_<topic>.md. Index in MEMORY.md.
  8. Commit, push, ampersand --pull on a different machine to verify portability.

Adding a new skill to the skills component

Two paths: NPM-installable skill (via npx skills add <github-repo>) or bundled (file copy).

For NPM: edit the apply block in install.sh's section 3. Add another npx --yes skills add <github-repo> invocation. Wrap in a non-fatal warn pattern.

For bundled: create the skill at claude/skills/<name>/SKILL.md in the repo, then add an install block following the component-gallery-reference pattern (mkdir + cp + ok).

Adding a new plugin to the config component

Edit claude/settings.json. Add "<plugin>@<marketplace>": true to enabledPlugins. If the marketplace isn't already known, add it to extraKnownMarketplaces with source and autoUpdate. Re-run install to merge, restart Claude Code to install.

For Yes& devs: bias toward enabling plugins the whole engineering team benefits from. Plugins that only one developer needs should live in their personal Claude Code config, not the shared dotfiles.

Adding a new CLAUDE.md rule

Edit claude/CLAUDE.md. Decide which existing section the rule belongs to (Memory Discipline, Verification Protocol, Design Work and Impeccable, Code Quality, etc.) - rules under the wrong heading get ignored. If the rule needs cross-team enforcement on projects where the dotfiles aren't installed, also drop it into a project-root CLAUDE.md template.
