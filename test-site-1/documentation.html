    Complete reference for Sidecoach - the design intelligence orchestration system built into Claude Code.

    Getting Started

    Installation

    Sidecoach ships with Claude Code. No additional installation needed. Open Claude Code and start using flows immediately.

    Your First Command

    The quickest way to understand Sidecoach is to run a shape flow on a feature you want to build:

    /sidecoach shape my-new-component

    This launches the design exploration phase. Follow the guidance - it will guide you through research, strategy, and approach selection before you write any code.

    Five Mandatory Workflow Gates

    1. Before any new feature: Run /sidecoach teach if no PRODUCT.md exists with real content
    2. Before implementing: Run /sidecoach shape to get the design plan
    3. After implementing: Run /sidecoach audit, /sidecoach critique, and /sidecoach polish
    4. Before shipping: Run /sidecoach harden
    5. Maintain project context: Keep PRODUCT.md and DESIGN.md up to date

    Project Files: PRODUCT.md and DESIGN.md

    Sidecoach reads two files from your project root:

    • PRODUCT.md - Brand identity, users, register, anti-references, strategic principles. Required for all design work.
    • DESIGN.md - Color tokens, typography, components, spacing. Optional but strongly recommended. Follow the Google design.md spec.

    If either is missing, run /sidecoach teach first. Sidecoach without project context produces generic output.

    Command Reference

    Research and Strategy Phase

    /sidecoach teach

    Guided setup. Creates PRODUCT.md and DESIGN.md via brand verification. Run this first if PRODUCT.md is empty or missing.

    /sidecoach shape <feature>

    Plan design approach before building. Runs exploration and rapid iteration. Outputs guidance, checklist, and reference artifacts.

    /sidecoach extract <target>

    Extract design tokens from existing implementation into DESIGN.md. Use when you have working code and need to formalize the system.

    /sidecoach live <target>

    Live browser iteration and real-time refinement. Open your page in a browser and make adjustments in real time while Sidecoach guides you.

    Implementation Phase

    /sidecoach craft <feature>

    Build new component from scratch. Covers tokens, implementation, and motion. Outputs a complete component with all states.

    /sidecoach layout <target>

    Restructure spatial relationships and visual hierarchy. Use when layout needs rework or optimization.

    /sidecoach typeset <target>

    Refine type system. Font pairing, scale, readability. Ensures typography aligns with your design tokens.

    /sidecoach animate <target>

    Implement production animation with exponential easing and reduced-motion support. Follows accessibility standards.

    /sidecoach colorize <target>

    Color refinement and palette application. Use tokens from your design system. No hardcoded values.

    /sidecoach delight <target>

    Add personality, micro-interactions, and joy. Subtle but memorable details.

    Polish and Refinement Phase

    /sidecoach polish <target>

    16-point tactical refinement. Concentric radius, optical alignment, scale-on-press, focus states, and more.

    /sidecoach bolder <target>

    Increase visual weight, contrast, and presence. Make design elements stand out.

    /sidecoach overdrive <target>

    Amplify design to maximum expressive effect. Use for hero sections or statement moments.

    /sidecoach quieter <target>

    Reduce noise and visual complexity. Simplify, mute colors, remove borders.

    /sidecoach distill <target>

    Extract and preserve only essential elements. Minimalist approach.

    /sidecoach clarify <target>

    Make design language explicit and unambiguous. Remove visual metaphor.

    QA and Validation Phase

    /sidecoach audit <target>

    5-dimension technical audit: accessibility (a11y), performance, theming, responsive design, anti-patterns. Outputs Lighthouse metrics and violations.

    /sidecoach critique <target>

    Independent design review. Heuristics, cognitive load, emotional journey. Uses Nielsen's 10 usability heuristics.

    /sidecoach optimize <target>

    Performance and efficiency improvements. Reduce bundle size, optimize images, improve LCP.

    /sidecoach harden <target>

    Production-readiness sweep. Error states, edge cases, i18n, a11y. Before shipping.

    /sidecoach adapt <target>

    Responsive design across all breakpoints. Mobile, tablet, desktop verification.

    /sidecoach list

    Show all available flows grouped by phase. Explore what is available.

    Best Practices

    Always Start With Brand

    Run /sidecoach teach first if PRODUCT.md doesn't exist. Your brand is the constraint that prevents hallucination. Everything else flows from it.

    Shape Before Craft

    Never skip the shape flow. It identifies the design challenge, uncovers constraints, and prevents wasted implementation work. The time spent shaping is recouped 5-10x during implementation.

    Use References, Not Imagination

    When Sidecoach offers reference artifacts (fonts, components, motion patterns), use them verbatim. Don't approximate or substitute. References are cited and verified - your imagination is not.

    Validate After Every Phase

    Run the QA flows after each phase:

    • After shape: Do you understand the design challenge?
    • After implementation: Is it accessible? Performant? On-brand?
    • After polish: Does it feel right?
    • Before shipping: Is it production-ready?

    Keep Project Files Fresh

    PRODUCT.md and DESIGN.md are your project's memory. Update them as your design system grows. Extract tokens regularly. This continuity enables Sidecoach to make smarter decisions across sessions.

    Use Soft-Fail Validation

    Validation runs on every flow but doesn't block you. A critical accessibility issue will be flagged - address it. A minor spacing suggestion will be noted - consider it. You are always in control.

    Compose Flows for Complex Work

    Sidecoach supports flow composition. For a major redesign, you might chain: shape - craft - audit - polish - adapt - harden. The system routes between flows intelligently based on your project state.

    Recommended Design Workflow

    1. Plan: /sidecoach shape my-feature - Understand the challenge
    2. Build: /sidecoach craft my-feature - Implement with tokens and motion
    3. Review: /sidecoach audit + /sidecoach critique - Technical and design review
    4. Polish: /sidecoach polish - Tactical refinement
    5. Test: /sidecoach adapt - Responsive and accessibility check
    6. Ship: /sidecoach harden - Production readiness

    Implementation Guidelines

    Use Design Tokens, Never Hardcode

    Every color, font size, spacing value should come from your design system. Hardcoded values are anti-patterns. Your DESIGN.md tokens are the single source of truth.

    Semantic HTML First

    Use semantic elements - <nav>, <button>, <article> - before styling. Semantics enable accessibility, SEO, and maintainability.

    Responsive by Default

    Design for mobile first. Use viewport units and media queries. Test at 320px, 768px, and 1200px minimum. Sidecoach's adapt flow verifies this.

    Component States Matter

    Every interactive component needs default, hover, focus, and active states. Animation states (enter, exit, loading). Error states. Accessibility states (aria-pressed, aria-expanded). Design all of them.

    Avoid Common Anti-patterns

    • No hardcoded colors - use CSS variables
    • No layout shift - CLS should be 0
    • No deprecated APIs - check console health
    • No unhandled focus - every element is keyboard navigable
    • No missing alt text - images have descriptions
    • No unmeasured font loading - fonts should be preloaded

    Validation and Quality Assurance

    5-Dimension Audit

    Every design runs through 5 dimensions of validation:

    1. Accessibility (a11y): WCAG 2.1 AA compliance, screen reader testing, keyboard navigation
    2. Performance: Lighthouse metrics, bundle size, LCP, CLS, FID
    3. Theming: CSS variable usage, no hardcoded values, semantic naming
    4. Responsive: Mobile, tablet, desktop rendering and interaction
    5. Anti-patterns: Console health, deprecated APIs, layout shifts, contrast violations

    Design Critique

    Independent design review using Nielsen's 10 usability heuristics:

    • System visibility and affordance
    • Match between system and real world
    • User control and freedom
    • Error prevention and recovery
    • Aesthetic and minimalist design
    • Cognitive load and clarity

    Responsive Design Standards

    Breakpoints

    Define breakpoints in your DESIGN.md. Standard breakpoints:

    • mobile: 320px - 767px
    • tablet: 768px - 1023px
    • desktop: 1024px+

    Touch Targets

    All interactive elements (buttons, links, toggles) must be at least 40x40 pixels. This ensures mobile usability and accessibility.

    Flexible Layouts

    Use flexbox and grid. Avoid fixed widths. Let content guide the layout. Test with text at 200% zoom.

    Image Responsiveness

    Use srcset and sizes attributes. Optimize for different pixel densities. Never serve desktop images to mobile.

    Motion Patterns and Animation

    Easing Functions

    Sidecoach enforces exponential easing for all motion. No linear animations. No cubic-bezier approximations.

    /* Good */
transition: transform 0.3s cubic-bezier(0.34, 1.56, 0.64, 1);

/* Bad */
transition: transform 0.3s linear;

    Reduced Motion Support

    Always respect prefers-reduced-motion:

    @media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}

    Standard Durations

    • 150ms - Micro-interactions (hover, focus states)
    • 300ms - Component transitions (menus, modals)
    • 500ms - Page transitions
    • 1000ms+ - Entrance animations

    Accessibility Standards

    WCAG 2.1 Level AA

    All designs must meet WCAG 2.1 Level AA. This is not optional. Sidecoach's audit flow verifies this automatically.

    Color Contrast

    Minimum 4.5:1 for normal text, 3:1 for large text (18px+ bold, 24px+ normal).

    Keyboard Navigation

    Every interactive element must be keyboard accessible. Tab order must be logical. Focus states must be visible.

    Screen Reader Support

    Semantic HTML first. ARIA labels only when needed. Test with VoiceOver (Mac) or NVDA (Windows).

    Form Accessibility

    Every input must have a label. Error messages must be associated with fields. Validation must happen on blur and submit.

    Component State Matrix

    Every component needs these states documented and tested:

    State Type States
    Interaction Default, hover, active, focus, disabled
    Data Empty, loading, error, success, data-present
    Responsive Mobile (320px), tablet (768px), desktop (1024px)
    Animation Enter, exit, loading spinner, transition

    Integration with Claude Code

    Slash Commands in Claude Code

    Type any Sidecoach command as a slash command in Claude Code:

    /sidecoach shape new-button-component

    Claude will route to the right flow and execute it with full context from your project's PRODUCT.md and DESIGN.md.

    Natural Language Routing

    You don't need to use the exact command. Sidecoach understands natural language:

    • "Check accessibility" routes to /sidecoach audit
    • "Make this responsive" routes to /sidecoach adapt
    • "Review the design" routes to /sidecoach critique
    • "I want motion" routes to /sidecoach animate

    Context Persistence

    Sidecoach maintains session memory. Your design decisions, validation results, and token references persist across commands. No context loss between flows.

    Frequently Asked Questions

    Do I need PRODUCT.md and DESIGN.md to use Sidecoach?

    Not initially. But Sidecoach will produce better guidance with them. Run /sidecoach teach to create PRODUCT.md. Extract or manually create DESIGN.md after your first implementation.

    Can I use Sidecoach without Claude Code?

    Sidecoach is built into Claude Code. It runs directly in your Claude Code session and uses your project context.

    What happens if my design doesn't meet validation gates?

    Validation is soft-fail by default. Critical issues (accessibility) are flagged - fix them. Minor suggestions are noted - consider them. You always retain control.

    Can I compose multiple flows into a workflow?

    Yes. Sidecoach supports flow composition and chaining. For a major redesign, you might run: shape - craft - audit - critique - polish - adapt - harden in sequence.

    How do I handle custom design tokens not in the reference?

    Add them to your DESIGN.md. Document your custom spacing scale, custom font sizes, custom colors. Sidecoach will use them in subsequent flows.

    What if Sidecoach suggests something I disagree with?

    You're in control. Flows output guidance, not commands. Read the guidance, consider it, and make your own decision. Sidecoach serves you, not the reverse.

    Can I disable validation?

    No. Validation runs soft-fail on every flow. The philosophy is transparency, not coercion. You see every issue and decide what to address.
