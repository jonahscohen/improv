# Improv Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

> **Scope note:** This is a large system with 8 phases. Each phase produces testable, working software. Phases 1-3 are the critical path (server, core, manipulate mode). Phases 4-8 build on that foundation. An agent executing this plan should complete one phase fully before starting the next.

> **Design spec:** `docs/superpowers/specs/2026-05-04-improv-design.md` - contains full architecture, all type definitions, component catalog, and prior art analysis.

> **Security note:** The toolbar renders a static HTML template into an isolated Shadow DOM with no user-supplied content. This is dev-only and reviewed as safe by the project owner.

**Goal:** Build a dotfiles component that enables in-browser visual micro-adjustments (manipulate, prompt, annotate, layout) and sends structured diffs to Claude Code via MCP.

**Architecture:** Three layers - a framework-agnostic core browser script (Shadow DOM overlay, Constructable Stylesheets preview, WebSocket transport), optional framework adapters (React/Vue/Svelte), and a local Node.js MCP server (source resolution, diff formatting, component scanning). Browser and server communicate via WebSocket JSON-RPC on localhost. Server and Claude Code communicate via MCP stdio.

**Tech Stack:** TypeScript (all layers), esbuild (browser bundle), Node.js MCP SDK, WebSocket (ws), @medv/finder (CSS selectors), vitest (testing)

**Existing patterns:** Follow the voice-output component pattern: server code in `claude/` dir, MCP server registered in `~/.claude.json` via Python JSON merge, skill at `claude/skills/improv/SKILL.md`, npm install at `~/.claude/improv/`.

### Task 3: WebSocket server with JSON-RPC protocol

**Files:**
- Create: `improv/server/ws-server.ts`
- Create: `improv/server/connection-manager.ts`
- Create: `improv/__tests__/server/ws-server.test.ts`

ConnectionManager: tracks WebSocket connections by ID, supports add/remove/get/getAll/findByWs/broadcast/size.

WsServer: wraps WebSocketServer from ws. Key behaviors:
- `start(preferredPort)` tries ports 9223-9232, returns first available
- New connections must send a handshake message (method: 'handshake') within 5 seconds or get closed with code 4001
- Non-handshake first messages close with code 4002
- After handshake, messages route to registered handlers via `onMessage(method, handler)`
- Handlers return values that become JSON-RPC responses
- `broadcastToClients(method, params)` pushes messages to all connected browsers
- `getConnections()` returns all active BrowserConnection objects
- `stop()` cleans up all timers and closes the server
